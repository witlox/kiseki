## Cross-verified expected signature
Three independent HMAC-SHA256 implementations (Python `hmac.new`,
`openssl dgst -sha256 -hmac`, Rust `aws-lc-rs`) all produce:
  ea21d6f05e96a897f6000a1a293f0a5bf0f92a00343409e820dce329ca6365ea

This value is what kiseki's `s3_auth::canonical_request` +
`derive_signing_key` chain emits for the `.creq` and `.sts`
shown above. It diverges from the value the AWS test-suite README
historically published (`5fa00fa31553b73e...`), which we believe
to be an upstream transcription error — the canonical-request
hash `bb579772...` MATCHES AWS-published, so the canonical-request
bytes are correct; only the final HMAC value in the README is
inconsistent with HMAC-SHA256 of the published intermediates.

The fixture-based test (rfc-aws-sigv4 suite) verifies kiseki's
canonical_request output equals the bytes in get-vanilla.creq
verbatim — that is the spec-anchored check.
